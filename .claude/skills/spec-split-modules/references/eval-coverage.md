# Eval（功能验证）覆盖策略

## 目标

每个模块的 e2e 验证文档要**能证明该模块交付达标**——既不漏检关键路径，也不过度覆盖尚未交付的能力。

## 一、覆盖矩阵

以下 8 个维度是**候选检查项**，不是每个模块都要全填。按模块类型选择相关维度：

| 维度 | 什么模块必测 | 什么模块可跳过 |
|---|---|---|
| 1. 编译/类型/构建 | 全部 | 无 |
| 2. 主 happy path | 全部有 UI 或有 API 的模块 | 纯类型定义模块 |
| 3. CRUD 全流程 | 承载领域实体的模块 | 纯展示模块 |
| 4. 过滤/搜索/分页 | 列表页 | 详情页 |
| 5. 表单校验/边界输入 | 有表单输入的模块 | 只读展示模块 |
| 6. 错误处理（4xx/5xx） | 涉及外部依赖的模块 | 无网络请求的模块 |
| 7. 空状态/加载态/终态 | 有异步数据的模块 | 静态组件 |
| 8. 跨模块联动（深链、跳转、事件） | 上层集成模块（如 M5） | 独立业务模块 |
| 9. 性能/A11y/无障碍 | 高交互频次页面 | 内部工具页面 |

## 二、区分"技术验证"与"功能验证"

**技术验证**（tech verification）：无 UI 或本模块尚不能独立跑完整功能时使用。
- 编译通过、导出符号数量、curl 联调 API、devtools 状态检查
- M1 基础设施是典型：无页面 → 只做技术验证

**功能验证**（e2e / behavior）：用户视角的操作步骤 + 期望结果。
- 有 UI 的模块以此为主
- 每个检查项包含：**操作步骤** + **Expected**（可观察的现象）

**判断依据**：模块交付后，是否能让最终用户"看到 / 用到"某个能力？能 → 功能验证优先；不能 → 技术验证优先。

## 三、检查项的写法

一条好的检查项满足：

1. **可勾选**：以 `- [ ]` 开头
2. **有前置条件**：需要什么数据、什么服务运行中，说清楚
3. **有具体操作**：不要写"验证 X 功能正常"，要写"点击 XX 按钮 → 输入 YY → 提交"
4. **有可观察的期望**：`Expected:` 后面是能用眼睛看到或用命令验证的结果，不是"应该正常工作"

**反例**（模糊）：
```
- [ ] 创建用例功能正常
```

**正例**（可执行）：
```
- [ ] **创建用例（上传文件）**
  - 点「创建用例」按钮 → Modal 弹出
  - 填 name = "test-case-1"，切到 Init Tab → 「上传文件」→ 拖入 200 行 shell
  - Expected: 上传成功 toast，自动切到「直接编辑」Tab，TextArea 显示脚本内容
  - 提交 → Expected: Modal 关闭，列表刷新，新用例出现在第一行
```

## 四、跨模块依赖的处理

某个测试项依赖尚未交付的下游模块（例：M2 里"测试用例被 M3 任务引用后返回 409"）：

- **标记 TODO**，不作为本模块交付条件
- 说明依赖：`需 M3 交付后测；M2 独立验收时跳过此项`
- 在被依赖模块（M3）的 e2e 里补上这条

## 五、章节结构模板

按模块的实际业务组织，通用骨架：

```markdown
# M{n} {模块名}模块验证文档

**对应规格**：`{path/to/spec}`
**前置**：{依赖模块已交付；服务运行中；数据准备等}

---

## 1. {核心场景 A} 例：创建用例

- [ ] **场景 1.1 具体操作**
  - Step ...
  - Expected: ...

## 2. {核心场景 B} 例：编辑用例
...

## 3. {边界与异常}
- [ ] 文件过大、必填缺失、并发冲突 ...

## 4. {空状态 / 终态}
- [ ] 无数据时显示 Empty
- [ ] 加载中显示 Skeleton

## 5. 交付验收清单
- [ ] 上述所有检查项通过
- [ ] `git status` 干净
- [ ] devtools 无 error/warning

---

**文档版本**：v1.0  |  **拆分自**：父规格 §{对应章节}
```

## 六、覆盖度自检

模块 e2e 写完后过一遍：

- [ ] 交付物清单里的每个文件，都至少被一条检查项覆盖到
- [ ] 每个"用户可见的能力"（父规格 §7 或类似章节里描述的 UI 场景），都有对应检查项
- [ ] 边界与异常至少有 3 条（过大输入 / 冲突 / 权限 / 网络失败等）
- [ ] 空状态 + 加载态 各 1 条
- [ ] 依赖 M{n-1} 的地方，明确写出"前置：M{n-1} 已交付"
- [ ] 没有依赖下游模块的检查项（若有，标 TODO）
