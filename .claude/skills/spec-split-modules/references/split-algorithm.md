# 模块拆分算法

## 目标

把一份大型 spec 设计文档，拆成若干粒度合理、职责清晰、可独立 spec → plan → implementation 的子模块。

拆分不是"把文档切成几段"，而是**从技术方案里识别出可以独立交付、独立验证的最小单元**。

## 一、拆分粒度原则

一个"合格"的模块，同时满足下列条件：

1. **单一职责**：一个模块只解决一件事，可用一句话概括（例：M1 搭技术地基、M2 做用例 CRUD、M5 做 Trace 深链）
2. **有独立交付物**：文件清单（组件、接口、类型、脚本）能列清楚，不与其他模块交叉
3. **可独立验证**：交付后有一组可执行的验收步骤，不依赖尚未交付的模块
4. **依赖 DAG 无环**：只允许"被依赖"，不允许循环
5. **规模适中**：一个模块的 spec 内容大约 150–350 行；小于 100 行说明过细，应合并；大于 500 行说明过粗，应再拆

## 二、拆分启发式

按以下优先级尝试拆分维度，选择**最符合父规格结构**的一种：

### 维度 A：按"页面 / 用户旅程"拆（前端类推荐）

若父规格有清晰的"路由 / 页面 / 用户操作流"章节，按页面拆是最直观的：
- 一个页面（或强内聚的一组页面）= 一个模块
- 页面内的组件、状态、API 归到同一个模块
- 例：M2 用例管理页 / M3 任务管理页 / M4 任务详情页

### 维度 B：按"业务能力 / 领域实体"拆（后端类推荐）

若父规格围绕多个领域实体展开（User、Order、Payment 等）：
- 一个实体的完整生命周期（CRUD + 状态机 + 关联查询）= 一个模块
- 跨实体的编排流程单列为"聚合服务"模块

### 维度 C：按"分层 / 依赖阶段"拆（复杂系统推荐）

- 底层地基（类型定义、公共库、基础设施）单独一层
- 中层能力（业务模块）
- 上层集成（联动、深链、跨模块流程）
- 例：M1（基础设施）→ M2/M3/M4（业务）→ M5（深链联动）

### 维度 D：按"实施阶段 / 交付节奏"拆（大型项目推荐）

若父规格已列出 Phase 1 / Phase 2 / Phase 3 的实施计划，直接沿用是最省事的：一个 Phase = 一个模块。前提是各 Phase 内聚。

**组合使用**：绝大多数场景是 A + C 或 B + C。先拆一层"基础设施 vs 业务模块"，再在业务层按 A 或 B 细分。

## 三、依赖建模

拆完后必须画出依赖 DAG，规则：

- 每个模块列出：**依赖谁**（我要用别人的产物）、**被谁依赖**（别人要用我的）
- 只允许强依赖走箭头。弱依赖（"如果 M2 交付了，M3 展示更完整，但没有也能跑"）标记为"推荐但非强制"
- DAG 无环。若发现循环，说明模块划分错误，需要重拆
- 推荐执行顺序 = DAG 拓扑排序（有多个合法顺序时，按父规格 Phase 或用户旅程排列）

**依赖表模板**：

| 模块 | 依赖 | 被依赖 | 关键接触点 |
|---|---|---|---|
| M1 基础设施 | 无 | M2、M3、M4、M5 | types、api、store、路由 |
| M2 用例管理 | M1 | (M3 弱依赖) | store/evalCase、api |
| M3 任务管理 | M1、(M2 弱) | M4 | store/evalTask、路由 |

## 四、模块识别的红旗信号

拆分完检视一遍，出现下列信号说明还有优化空间：

| 红旗 | 含义 | 处理 |
|---|---|---|
| 两个模块都在改同一个文件 | 边界模糊 | 合并、或把公共文件下沉到基础设施模块 |
| 一个模块只有 1 个交付文件 | 粒度过细 | 合并到相邻模块 |
| 一个模块横跨多个页面/多个领域 | 粒度过粗 | 再切一刀 |
| 模块名叫"其他" / "杂项" | 边界没想清楚 | 重新命名或再拆 |
| 依赖关系解释不清 | 抽象层次乱了 | 重新按维度 C 分层 |
| 循环依赖 | 违反 DAG | 抽出公共依赖成新模块 |

## 五、命名约定

- 模块编号：`M1 / M2 / ...`（顺序 = 推荐执行顺序）
- 模块名：2-6 字中文，动宾或名词短语（"基础设施"、"用例管理"、"Trace 深链"）
- 文件名：`M{n}-{kebab-english}-spec.md` / `M{n}-{kebab-english}-e2e.md`
  - 例：`M1-infrastructure-spec.md`、`M5-trace-deeplink-e2e.md`

## 六、执行清单

拆分时按顺序走：

1. [ ] 通读父规格，画出章节树
2. [ ] 识别"分层"（基础设施 vs 业务 vs 集成）
3. [ ] 每一层按维度 A/B/C/D 再细分
4. [ ] 列出每个模块的一句话职责
5. [ ] 列出每个模块的交付物清单（预估文件、行数）
6. [ ] 画依赖 DAG
7. [ ] 用"红旗信号表"复查一遍
8. [ ] 输出模块列表（README 索引）
