## 2. 技术要点

### 2.1 字段命名一致性
- 所有类型定义与 API 层保持 snake_case，与后端 DTO 1:1 映射
- 不做驼峰转换，避免前后端字段名不一致风险

### 2.2 竞态处理
- 三个 Store 均使用模块级 AbortController 处理并发请求
- 每次 fetch 前 abort 上次请求，避免旧请求覆盖新请求结果

### 2.3 智能轮询
- Task Store: 5 秒轮询，仅当存在 PENDING/RUNNING 任务时继续
- Instance Store: 3 秒轮询，仅当存在 PENDING/RUNNING 实例时继续
- 静默刷新（不设置 loading=true），避免表格闪烁
- 全部终态时自动停止轮询，节省带宽

### 2.4 类型安全
- 使用 TypeScript strict mode
- 所有 API 返回值明确类型标注
- ListPage<T> 泛型支持类型复用

## 5. 关键决策

### 5.1 轮询策略
- 选择智能轮询而非 SSE：后端无 SSE 支持，轮询为唯一实时方案
- 5s/3s 间隔平衡实时性与后端压力
- 终态自动停止避免无效请求

### 5.2 状态管理
- 选择 Zustand 而非 Redux：与现有模块一致，轻量级，无 boilerplate
- 模块级 AbortController 而非 state 内：避免触发不必要的 re-render

### 5.3 目录结构
- 按功能分组（Cases / Tasks / TaskDetail）而非按组件类型
- 与现有 pages/Trace/ 对齐
